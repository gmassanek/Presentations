MVC for the client?

  • Mehhhhh
  • More like widgets/partials/pieces of a site responsible for
    • Knowing about data (and how to get it)
    • Rendering itself
    • Rich interaction with the user
  • Pushes browser-y stuff to the browser, lets the server do server-y stuff

How the CS App used to work

  • Basic rails call stack
    • Controller
    • Models
    • Groupon Universe (orders, deal, user, API)
    • Back through the controller
    • Rails view to render everything
    • Big packet down to client

How the cs app works with backbone

  • Basic rails call stack
    • Controller
    • Small rails view to render frame
    • Small packet down to client
    • Independent bb calls to cs api
      • Api controller
      • Models
      • Groupon universe (orders, deal, user, api)
      • Back through the controller
      • Presenter
      • JSON back to client

Our Backbone Setup - Widget Style

  • Router (identifies URL, calls controller)
  • Controller (retrieves data, renders view)
  • View - WIDGET
    • Accepts data (model)
    • Binds data to template to render HTML
    • Sets up events (with BB or vent), well scoped
    • Renders sub-views

Controller changes

Work in progress, but, we've been changing our controllers to do three tasks.
  • Delegate to retrievers that parse the incoming params and fetch corresponding data
  • Present the data
  • Authentication

Presenters, Templates, and Facile

  • Presenters. Accepts a set of data and prepares it for BB to use with Facile (and events)
  • Templates. LOGICLESS!
  • Facile. Binds data to templates using classes and ids
